01 Extraction Triggers
🔚
session_end Event
Primary trigger. Fired when user disconnects, timeout occurs, or explicit close. Most reliable signal.
ON: disconnect | timeout_15m | explicit_close
🔢
N-Turn Threshold
Trigger mid-session after every 20 turns for long interactions. Prevents memory loss on crash. Incremental extraction.
ON: turns_count % 20 == 0
⏱️
Time-Based Flush
Safety net. Catches any sessions that never fired a clean close event. Background cron every 2 hours.
ON: cron 0 */2 * * * · stale_sessions
02 The Extraction Prompt (LLM Worker)
extraction_worker.system_prompt SYSTEM PROMPT · ASYNC WORKER
# ══════════════════════════════════════════════════════ # ROLE # ══════════════════════════════════════════════════════ You are a precision memory extraction engine. Your sole job is to analyze a customer support chat transcript and output structured JSON. You do not summarize. You do not paraphrase loosely. You extract exact, verifiable, durable signals from human language.
# ══════════════════════════════════════════════════════ # EXTRACTION TAXONOMY # ══════════════════════════════════════════════════════ Extract ONLY the following six types. If no instance of a type exists, return an empty array for that key. Never hallucinate.
1. FACT → immutable, verifiable entity stated by the user 2. PREFERENCE → stated or strongly inferred behavioral preference 3. EPISODIC → what happened in this session (outcome, frustration, resolution) 4. PROJECT → any active goal, task, or initiative the user mentioned 5. SENTIMENT → emotional state arc across session (start → peak → end) 6. CRM_EVENT → billable or escalation-worthy event (complaint, churn_risk, upsell)
# ══════════════════════════════════════════════════════ # CONSTRAINTS — NEVER VIOLATE # ══════════════════════════════════════════════════════ - confidence below 0.60 → DO NOT extract, discard - do not infer facts from agent statements, only from user utterances - source_quote must be ≤ 120 chars, verbatim from transcript - if a fact conflicts with a prior known fact, set conflict: true - output ONLY the JSON object. No preamble, no explanation, no markdown fences
# ══════════════════════════════════════════════════════ # OUTPUT FORMAT # ══════════════════════════════════════════════════════ Return exactly this structure. All fields required unless marked optional. → See: extraction_output.schema.json (Section 03)

05 Full Execution Flow
01
User chat turn received
Append to MongoDB sessions.{session_id}.turns[] · update hot KV · increment turn_count
WRITE → Mongo
02
Trigger fires (session_end | N-turns | cron)
Enqueue extraction job · pass session_id, user_id, org_id · mark session as "processing"
READ → Mongo
03
LLM extraction worker runs
Pull full transcript from Mongo · send to LLM with system prompt · receive canonical JSON · validate schema · reject if malformed
READ → Mongo
04
Consolidation check (facts + preferences)
For each FACT + PREFERENCE: query PG for existing record by (user_id, key) · if conflict=true → apply recency-wins · set overridden_by FK · log to audit
READ/WRITE → PG
05
Write episodic + project memories to Mongo
Insert memory_entries documents · set embed_this=true · link back to extraction_id and session_id
WRITE → Mongo
06
Embed episodic + project entries → pgvector
For all entries where embed_this=true · generate embedding (text-embedding-3-large) · INSERT into memories_embeddings with user_id, session_date, mongo_memory_id
WRITE → pgvector
07
Write CRM events + sentiment to PG audit
INSERT crm_events · UPDATE session_audit with sentiment arc · trigger CRM webhook if action_required=true
WRITE → PG
08
Nightly batch: regenerate user profile.md
Compile {user_id}_profile.md from PG facts + preferences + active projects · store in /profiles/ · loaded at next session start with zero vector search cost
WRITE → MDREAD → PG

07 GDPR / Right to be Forgotten — Cascade Delete
⚠️ Delete Sequence (strict order)
  • PG: SET users.status = 'deletion_requested'
  • PG: SELECT all session_ids WHERE user_id = $1
  • Mongo: deleteMany({ session_id: { $in: session_ids }})
  • Mongo: deleteMany({ user_id: $1 }) // memory_entries
  • pgvector: DELETE WHERE user_id = $1 // embeddings
  • PG: DELETE user_facts, user_preferences, crm_events
  • PG: DELETE users WHERE id = $1
  • FS: rm -f /profiles/{user_id}_profile.md
  • PG: INSERT deletion_audit log with timestamp
🔐 Namespace Isolation (multi-tenant)
  • Every query MUST include WHERE org_id = $org AND user_id = $uid
  • pgvector: filter by user_id BEFORE cosine scoring
  • Mongo: compound index on (org_id, user_id, session_id)
  • PG RLS: enable row_security on all user tables
  • PG: CREATE POLICY per_user ON user_facts USING (user_id = current_user_id())
  • Profile MD: stored at /profiles/{org_id}/{user_id}.md
  • IAM: service account per org_id, no cross-tenant tokens

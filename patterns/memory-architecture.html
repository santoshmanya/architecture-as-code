▸ Enterprise Agent Memory · Customer Support Pattern

Episodic Memory Architecture

Extraction → Storage → RAG Retrieval → Pre-prompt Injection · MongoDB + PostgreSQL + pgvector + Markdown

Storage Layer
MongoDB
// session · episodic · raw
Raw session documents, chat history, extracted memory blobs. Flexible schema for evolving preference structures. Primary write target during live chat.
PostgreSQL
// facts · preferences · audit
Structured user facts, preference records, consolidation log, and audit trail. Source of truth for hard entities like account_id, location, policy tier.
pgvector
// semantic search · RAG
Embeddings of extracted memories and episodic reflections. Enables "remind me what we discussed" semantic recall with cosine similarity queries.
Markdown Files
// persona · config · prompts
Agent persona definitions, system prompt templates, consolidation rules, and per-tenant configuration. Version-controlled. Loaded at agent init.
Processing Pipeline
PHASE 01 Live Session SYNC · IN-CHAT
📥
Session Hydration MONGOPG
On conversation start → fetch user_id facts from PG + last 3 session summaries from Mongo → inject into system prompt
💬
Raw Turn Logging MONGO
Every turn appended to sessions.{session_id}.turns[] · includes role, content, timestamp, intent_tag
🔑
Hot KV Store MONGO
In-session key-value map: { "account_id": "ACC-001", "issue": "billing", "frustration_level": "high" } · agent reads this every turn
PHASE 02 Post-Chat Extraction ASYNC · BACKGROUND
🤖
LLM Extraction Job
Triggered on session_end event · LLM reads full transcript · extracts FACTS, PREFERENCES, EPISODIC_REFLECTIONS as structured JSON
⚖️
Consolidation Check PG
Compare extracted facts vs existing records · detect conflicts (e.g. diet: vegetarian → meat-eater) · apply recency-wins rule · log delta in audit table
🗄️
Write to Stores PGMONGOpgvector
Facts → PG user_facts · Episodic reflections → Mongo memory_entries · Embed reflection text → pgvector memories_embeddings
PHASE 03 Pre-Prompt Injection SYNC · CHAT START
🔍
Semantic Recall pgvector
Embed current user query → cosine similarity search → retrieve top-3 relevant past memories with session_date, summary
📋
Context Assembly PGMD
Merge: agent persona (MD) + hard facts (PG) + top preferences (PG) + recalled memories (pgvector) → inject as structured system prompt block

🧮

🎯 Retrieval Query
  • Filter strictly by user_id (no cross-tenant bleed)
  • Score threshold > 0.75 to avoid noise
  • Return: content_preview + session_date + memory_type
  • Join back to Mongo for full episodic detail if needed
💡 Reminder Phrasing
  • "On March 14th, you mentioned you prefer email contact"
  • "Last time we spoke about your Wi-Fi issue — that was resolved"
  • "You've previously told us you're based in Singapore"
  • Only surface if similarity score > 0.82 to avoid false recalls
Pre-Prompt Injection Template
# ── AGENT PERSONA ───────────────────────────────────── {{ load: personas/customer-support.md }} # ── KNOWN USER FACTS (from PG) ──────────────────────── user_id: {{ user.id }} account_tier: {{ user.facts.account_tier }} location: {{ user.facts.location }} language_pref: {{ user.facts.language }} # ── ACTIVE PREFERENCES (from PG) ────────────────────── contact_preference: {{ user.prefs.contact_pref }} friction_tolerance: {{ user.prefs.friction_tolerance }} # ── RECALLED MEMORIES (from pgvector, top-3) ────────── relevant_past_interactions: {% for memory in recalled_memories %} - date: {{ memory.session_date | format_date }} summary: {{ memory.content_preview }} type: {{ memory.memory_type }} {% endfor %} # ── INSTRUCTION ─────────────────────────────────────── If any recalled memory is directly relevant to the user's current message, naturally reference it. E.g. "On [date], you mentioned..." Do not surface all memories — only reference if relevance score > 0.82
Preference Consolidation (Conflict Resolution)
When the async extraction job detects a conflicting preference, apply recency-wins and preserve audit trail:

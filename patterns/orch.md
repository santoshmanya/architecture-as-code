Here's how each pattern maps to your three-tier hierarchy and when to reach for each.
Pattern 1 — Multi-Agent Router is your default entry pattern. The Super Agent is the only layer that sees the raw user input. It performs intent classification and hands off to the right Domain Agent without attempting to solve anything itself. This prevents foundational drift — a stateless agent getting passed a journey-level problem it has no context window or state management to handle. The routing decision should be deterministic and cheap; the Super Agent should not be doing multi-step reasoning here.
Pattern 2 — Orchestrator-Workers is what happens inside a Domain Agent for any complex end-to-end transaction. The Domain Agent is the sole owner of journey state — it knows where the user is in the flow, what's been completed, and what's blocking. It decomposes the goal into atomic subtasks and fans out to Foundational Agents in sequence or parallel. Foundational Agents stay dumb-and-functional: they receive a clean input, return a clean output, and retain nothing. The Domain Agent is the only entity that stitches results together.
Pattern 3 — Cooperative Swarm handles cross-domain journeys like a relocation that triggers both a real estate and a tax workflow. Rather than force all logic into one Domain Agent, the Super Agent facilitates a clean handoff between two Domain Agents. Each agent runs its own orchestration loop. The risk to design around is shared state — both agents need to agree on what the "source of truth" context object looks like at handoff time, otherwise you get data drift.
Pattern 4 — Self-Correction Loop is your safety net for high-stakes domains. After the Domain Agent completes a transaction, a dedicated Validator Foundational Agent runs a stateless business-rules check against the output. If it fails, the state object is returned to the Domain Agent for correction. This keeps the validation concern fully decoupled — the Validator knows nothing about the journey, only about the rules. Important for anything in finance, compliance, or legal where "complete" is not the same as "correct."
The practical assembly for most journeys is 1 → 2 → 4: Super Agent routes, Domain Agent orchestrates workers, Validator audits the result. Pattern 3 extends that when a single journey crosses domain boundaries

Why ADK is the preferred backbone (as you've already landed on): ADK gives you SequentialAgent, ParallelAgent, and LoopAgent as first-class primitives — your four patterns map almost 1:1 without you building the orchestration logic yourself. A2A protocol compliance is native. LangGraph requires you to hand-build these patterns via graph topology, which gives more control but more surface area to get wrong.
Where LangGraph wins: Complex journey state machines where you need fine-grained control over which node runs under which condition — e.g. a mortgage journey where the next step depends on a combination of credit score + doc status + user tier. LangGraph's conditional_edge with a full state dict is cleaner for that than ADK's agent composition.

The runtime adapter reads the manifest and instantiates the right ADK agent class or LangGraph node. This is what makes your factory reusable across Mango Bank, Agent Village, and EA Agents without duplicating orchestration code.
